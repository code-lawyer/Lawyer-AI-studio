# 决策记录

## 2026-03-23

### 决策 1：MVP 先做 Web 案件工作台，不做聊天壳
- 前端以案件工作台为中心，展示案件材料、任务、产物和审核状态。
- 不把首版交互建立在聊天窗口上，避免偏离已确认的产品形态。

### 决策 2：MVP 后端选用 FastAPI + SQLite
- 仓库当前没有现成应用壳和数据库层，MVP 需要先落一个低门槛、可运行的后端。
- SQLite 可以在不增加外部依赖的情况下完成数据库迁移，并满足当前原型阶段的持久化需求。
- Postgres 放到下一阶段，作为团队多人协作的升级项。

### 决策 3：外部执行器暂以本地适配器模拟
- 当前重点是先打通案件工作台闭环，而不是一次性接完真实外部执行环境。
- 本地适配器保留任务状态、日志、外部任务 ID 等接口形态，为后续替换真实 Claude Code 类执行器留边界。

### 决策 4：优先补运行记录和审核记录
- 相比继续增加 Agent 能力，当前阶段更缺的是可追踪性。
- 因此优先实现 `WorkflowRun`、`ReviewRecord`、文档预览、任务日志和导出链路。

### 决策 5：执行器协议先抽象成可替换层
- 真实 Claude Code 类执行环境尚未接入，但接口边界需要先稳定。
- 因此后端先定义统一执行器协议，支持 `submit`、`sync`、`cancel` 三个动作。
- `local_adapter` 作为 fallback，`external_stub` 和 `external_stub_hold` 作为外部执行模拟器，用来先验证状态流转和 UI 行为。

### 决策 6：明确 Claude Code 不是产品本体
- SuitAgent Workspace 的产品本体是案件工作台、领域数据、工作流和审核导出体系。
- Claude Code 仅作为可替换的 AI 执行器接入。
- 后续所有产品和开发决策，都以“案件工作台优先、执行器可替换”为前提，避免产品退化成聊天套壳。

### 决策 7：真实执行器先落在外部命令适配层
- 当前机器存在 `claude.exe`，但未完成登录认证，不能直接稳定跑通真实任务。
- 因此先实现通用外部命令执行器 `command_runner`，再在此基础上提供 `claude_cli` 模式。
- 这样既能把“真实外部调用”能力落进系统，又不会因为当前认证状态阻塞整体开发。
