# SuitAgent Workspace 开发计划

## 1. 总体技术方案

### 1.1 总体形态
SuitAgent Workspace 采用“Web 工作台 + 后端服务 + 可替换执行器”的架构。

产品由三层组成：
- 前端工作台：案件、文档、任务、产物、审核的统一界面
- 后端业务层：案件数据、工作流运行记录、导出和审核管理
- 执行器适配层：对接 Claude Code 类产品或其他 AI 执行环境

### 1.2 核心原则
- 人在回路，正式产物默认需要人工审核
- 单案件闭环优先于全局运营能力
- 规则先代码化，不继续堆纯文档工作流
- Claude Code 是执行层，不是产品本体
- 执行层必须可替换

### 1.3 技术栈
- 前端：Next.js
- 后端：FastAPI
- 数据库：V1 先用 SQLite，下一阶段升级 Postgres
- 队列：下一阶段引入 Redis + Celery 或 RQ
- 文件存储：本地磁盘起步，后续兼容对象存储
- 文档处理：复用 `.claude/tools/*.py`
- AI 执行层：Claude Code 类执行器适配层

## 3. 模块拆分

### 3.1 Case Service
职责：
- 创建案件
- 查询案件列表和详情
- 初始化案件目录
- 维护案件状态和时间线

### 3.2 Document Service
职责：
- 上传文档
- 存储原始文件
- 生成提取文本
- 提供预览
- 后续接 OCR / PDF / DOCX 处理链路

### 3.3 Workflow / Task Service
职责：
- 创建 `WorkflowRun`
- 创建和跟踪任务
- 管理任务状态、日志、重试、取消

### 3.4 Artifact Service
职责：
- 保存 AI 输出
- 管理审核状态
- 保存审核记录
- 导出 Word / Markdown

### 3.5 Agent Adapter Service
职责：
- 统一管理外部执行器接入
- 维护任务提交、状态同步、取消、结果回传

这层是 Claude Code 接入的真正落点。

### 3.6 Auth / Admin Service
职责：
- 登录鉴权
- 角色权限
- 模板与配置管理

## 4. Claude Code 接入策略

### 4.1 定位
Claude Code 在系统中只承担执行器角色，不承担以下职责：
- 不承担前端主界面
- 不承担案件组织
- 不承担审核和导出记录
- 不直接成为最终用户交互入口

### 4.2 适配层职责
执行器适配层必须提供统一接口：
- `submit(task_payload)`
- `sync(task_state)`
- `cancel(task_state)`

后续真实接入 Claude Code 时，需要补齐：
- 任务提交协议
- 状态轮询或回调协议
- 日志回传
- 结果文件落地

### 4.3 当前策略
当前已完成：
- `local_adapter`
- `external_stub`
- `external_stub_hold`
- `command_runner`
- `claude_cli`

下一步目标：
- 将 `claude_cli` 接到真实可用的认证环境
- 保留 `local_adapter` 作为 fallback

## 6. 三阶段排期

### Phase 1：MVP
目标：
- 做出案件工作台最小闭环
- 完成基础数据结构和状态流转
- 把 Claude Code 预留在执行器适配层

当前已完成：
- 案件列表与案件工作台前端
- 文档上传与提取预览
- `WorkflowRun` 与 `ReviewRecord`
- 任务日志、重试、取消
- 产物审核与导出
- SQLite 持久化
- 执行器协议与 stub 模式

### Phase 2：Beta
目标：
- 接入真实外部执行器
- 提升团队协作能力

任务：
- 接入真实 Claude Code 类执行器
- 增加状态轮询和回调链路
- 增加权限体系
- 增加模板配置页
- 增加更多文档处理链路

### Phase 3：Production Hardening
目标：
- 提升稳定性与可运维性

任务：
- SQLite 升级 Postgres
- 引入 Redis / 队列
- 完善监控与错误告警
- 增加端到端测试和回归测试

## 8. 风险与依赖

### 8.1 依赖
- Claude Code 类执行环境接入方式
- 本地文档处理工具可用性
- Word 导出链路稳定性

### 8.2 风险
- 如果执行器适配层设计过薄，后续容易重新耦合到 Claude Code 宿主
- 如果后续过度强调聊天交互，产品可能偏离案件工作台定位
- 如果没有回调或轮询协议，真实外部执行器难以稳定接入

## 9. 首批下一步任务

### 9.1 第一优先级
- 设计真实 Claude Code 类执行器的输入输出协议
- 实现一个最小可运行的外部执行器适配器
- 将 `external_stub` 替换成真实链路

### 9.2 第二优先级
- 为执行器增加日志回传与结果回传
- 在前端展示更明确的执行器状态和来源

### 9.3 第三优先级
- 增加基础登录与角色权限
- 规划 Postgres 与 Redis 迁移

## 10. 当前结论
接下来的开发重点不是继续堆 UI，而是把“真实 Claude Code 类执行器接入”做实，同时保持 SuitAgent Workspace 作为产品本体不被执行器反向绑架。
