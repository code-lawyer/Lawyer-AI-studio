## 路由机制概述

主Agent在接收到用户需求时，应首先进行需求分析，判断是单一功能需求还是复合工作流需求：

- **单一功能需求**: 直接路由到对应专门Agent
- **复合工作流需求**: 制定工作流组合，按顺序调用多个Agent

## 路由执行流程

### 主Agent路由决策树

```mermaid
flowchart TD
    A[用户输入需求] --> B{检测到单一功能触发词?}

    B -->|是| C[识别Agent类型]
    B -->|否| D{检测到复合场景触发词?}

    C --> E[立即调用对应Agent]
    E --> F[Agent完成任务]
    F --> G[返回结果给用户]

    D -->|是| H[启动预设工作流]
    D -->|否| I[智能分析需求]

    H --> J[按顺序调用多Agent]
    J --> K[整合所有Agent结果]
    K --> L[调用Reporter生成综合报告]
    L --> G

    I --> M[询问用户具体需求]
    M --> N[根据用户反馈定制工作流]
    N --> J
```

### 路由优先级规则

1. **关键词匹配优先**: 精确匹配触发词 → 立即路由
2. **场景识别优先**: 匹配复合场景 → 启动预设工作流
3. **智能分析最后**: 无法自动识别 → 询问用户并智能分析

## Agent调用标准

### 单一任务调用模式
```
用户: "写起诉状"
主Agent: 检测到"写起诉状" → 路由到Writer Agent
Writer Agent: 完成起诉状起草 → 返回结果
主Agent: 将结果返回用户
```

### 复合任务调用模式
```
用户: "收到起诉状需要应诉"
主Agent: 检测到复合场景 → 启动预设工作流
主Agent: 按顺序调用 Agent1 → Agent2 → Agent3...
Reporter Agent: 整合所有结果 → 生成综合报告
主Agent: 将最终报告返回用户
```

## 工作流执行原则

### 核心原则

1. **自动触发机制**
   每个Agent在完成自身工作后，必须自动触发后续Agent，而不是等待手动调用。
2. **上下文继承**
   启用上下文继承，保持分析一致性。
3. **增量更新**
   复用历史分析结果，提高效率。
4. **质量保证**
   每个Agent都具备内嵌验证或专项审查机制。

### 工作流完整性保障

- **自动触发**：✅ 启用
- **上下文维护**：✅ 启用
- **质量检查**：✅ 启用
- **增量支持**：✅ 启用
