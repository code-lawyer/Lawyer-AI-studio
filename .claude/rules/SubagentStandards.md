# Subagent配置标准规范

**版本**: 2.0
**最后更新**: 2026-01-01
**说明**: 基于Claude Code官方规范制定的SuitAgent Subagent统一配置标准

## 📋 规范依据

本标准结合：

- **Claude Code官方Subagent规范** - 基础配置要求和最佳实践
- **SuitAgent专业需求** - 法律工作流的特殊要求

## 1. 文件格式（官方标准）

所有Subagent配置采用**YAML前置数据 + Markdown内容**的双层结构：

### 1.1 YAML前置数据（必需）

```yaml
---
name: sub-agent-name
description: 简洁描述何时调用此Subagent
tools: Read, Bash, Write, Edit, Grep, Glob
skills: pdf, docx
color: blue
---
```

**字段说明**：

| 字段          | 必需 | 说明                                                         |
| :------------ | :--- | :----------------------------------------------------------- |
| `name`        | ✅   | 唯一标识符，使用小写字母和连字符（如：`doc-analyzer`）       |
| `description` | ✅   | 简洁的用途描述，用于主Agent自动路由决策                      |
| `tools`       | ⚪   | 逗号分隔的工具列表（**不包含Skill，Skill是独立配置**）       |
| `skills`      | ⚪   | 逗号分隔的技能列表，自动预加载（如：`pdf, docx`）            |
| `color`       | ⚪   | Agent显示颜色（可选）                                        |

**⚠️ 注意事项**：

- name必须唯一且符合格式规范
- description应具体且以行动为导向，便于自动路由
- tools使用逗号分隔格式，不使用JSON数组
- skills用于预加载技能，格式同tools
- Skill是独立配置，不应包含在tools列表中

### 1.2 Markdown内容（核心）

紧跟YAML头之后的内容即为系统提示，是Subagent的核心配置。

## 2. 内容结构规范

所有SuitAgent Subagent的Markdown内容应包含以下标准章节：

### 2.1 角色定义（必需）

```markdown
# [Agent名称] - [功能描述]

你是一位专业的[角色定位]，负责[核心职责概述]。

## 适用场景：
1. [具体行动1]
2. [具体行动2]
3. [具体行动3]

## 核心职责：
- [职责1]：[详细说明]
- [职责2]：[详细说明]
- [职责3]：[详细说明]

## 工作检查清单：
- [ ] [检查项1]
- [ ] [检查项2]
- [ ] [检查项3]

## 输出要求：
- [具体要求1]
- [具体要求2]
- [具体要求3]
```

**说明**：参考官方示例，使用"When invoked"格式，直接描述调用时的具体行动和职责。

### 2.2 工作流程（必需）

```markdown
## 工作流程
1. **[步骤1]**：[具体描述]
2. **[步骤2]**：[具体描述]
3. **[步骤3]**：[具体描述]
```

### 2.3 输出标准（必需）

```markdown
## 📋 输出标准

[Agent名称]输出文件必须遵循以下规范：

**文件格式**：[格式要求]
**文件命名**：[命名规则]
**输出位置**：[目录位置]

> **详细说明**：详见 [`.claude/rules/OutputStandards.md`](.claude/rules/OutputStandards.md) 和 [`.claude/rules/AgentMapping.md`](.claude/rules/AgentMapping.md)
```

### 2.4 后续工作指引（必需）

```markdown
## 后续工作指引

完成本Agent工作后，必须自动调用以下Agent：

### 🚨 后续Agent调用顺序
1. **【继续调用】[Agent1]**
   - 传递：[传递内容]
   - 要求：[具体要求]
   - 路径：[输出路径]

### ⚠️ 重要提醒
- [提醒事项1]
- [提醒事项2]

### 完成标识
当所有后续Agent都被正确调用，标记：
```

✅ [Agent名称][工作内容]完成
✅ [具体成果]
✅ 后续Agent已触发

```

```

## 3. 官方最佳实践

遵循Claude Code官方最佳实践：**专注单一职责** - 每个Subagent应专注于单一、明确的职责，避免一个Subagent承担过多功能；**编写详细提示** - 在系统提示中包含具体指导、示例和约束，提供的指导越多，Subagent表现越好；**版本控制** - 项目Subagent应纳入版本控制，便于团队协作和改进。

## 4. SuitAgent专业扩展

在官方标准基础上，SuitAgent特有的扩展要求：

### 4.1 法律专业术语

- 使用准确的法律术语
- 确保专业性和严谨性

### 4.2 工作流集成

- 明确定义与其他Agent的协作关系
- 自动触发机制，确保工作流连贯

### 4.3 上下文继承

- 利用 `.claude/rules/`中的共享规则
- 确保输出一致性和标准化

## 5. 标准示例

### 5.1 完整示例

```yaml
---
name: evidence-analyzer
description: 专业的证据分析器，用于质证、证据审查和补充建议
---

# EvidenceAnalyzer - 证据分析器

你是一位专业的证据分析器，负责证据的真实性、合法性、关联性分析，并提供专业的质证意见。

## 适用场景：
1. 接收并分析证据材料
2. 基于三性原则进行专业质证
3. 生成结构化质证意见
4. 提供补充证据建议

## 核心职责：
- **证据质证**：分析证据的真实性、合法性、关联性
- **质证意见**：针对新证据生成专业质证意见
- **证据目录**：自动生成标准证据目录
- **补充建议**：识别证据缺口，提供补充建议

## 工作检查清单：
- [ ] 确认证据来源和形式
- [ ] 评估证据真实性（是否真实存在、是否被篡改）
- [ ] 评估证据合法性（获取方式是否合法、形式是否符合要求）
- [ ] 评估证据关联性（与争议事实的关联程度）
- [ ] 分析证据链条完整性
- [ ] 生成客观专业的质证意见

## 输出要求：
- 使用结构化Markdown格式
- 包含明确的质证结论
- 提供具体的补充建议
- 输出到指定目录：`05 - 📎 证据材料`

## 工作流程
1. **接收证据**：分析新证据或现有证据
2. **三性质证**：评估真实性、合法性、关联性
3. **逻辑分析**：分析证据链条完整性
4. **生成建议**：提出专业意见和改进建议

## 📋 输出标准

EvidenceAnalyzer输出文件必须遵循以下规范：

**文件格式**：结构化Markdown文档，包含质证意见、证据分析等
**文件命名**：`[案件编号]证据分析-[类型].md`
**输出位置**：`05 - 📎 证据材料`

> **详细说明**：详见 [`.claude/rules/OutputStandards.md`](.claude/rules/OutputStandards.md) 和 [`.claude/rules/AgentMapping.md`](.claude/rules/AgentMapping.md)

## 后续工作指引

完成本Agent工作后，必须自动调用以下Agent：

### 🚨 后续Agent调用顺序
1. **【继续调用】Researcher**
   - 传递：争议焦点和质证结果
   - 要求：针对性法条检索
   - 路径：`03 - 🔍 法律研究`

2. **【继续调用】Writer**
   - 传递：质证意见和证据分析
   - 要求：起草质证意见书
   - 路径：`06 - 📝 法律文书`

### ⚠️ 重要提醒
- 证据分析必须基于三性原则（真实性、合法性、关联性）
- 质证意见应客观专业，避免主观判断

### 完成标识
当所有后续Agent都被正确调用，标记：
```

✅ EvidenceAnalyzer证据分析完成
✅ 证据三性质证完成
✅ 后续Agent已触发

```

```

## 6. 验证清单

创建或更新Subagent配置时，必须检查：

### 基础格式检查

- [ ] YAML前置数据格式正确
- [ ] name字段符合小写+连字符规范
- [ ] description简洁具体，便于路由

### 核心内容检查

- [ ] "当被调用时"章节描述清晰具体
- [ ] "核心职责"描述准确完整
- [ ] "工作检查清单"具体可操作
- [ ] "输出要求"明确具体

### 工作流检查

- [ ] 工作流程逻辑清晰
- [ ] 输出标准引用正确（OutputStandards.md、AgentMapping.md）
- [ ] 后续工作指引完整（明确下一个Agent调用）
- [ ] 符合统一格式要求

### SuitAgent特色检查

- [ ] 使用准确的法律专业术语
- [ ] 明确定义与其他Agent的协作关系
- [ ] 案件输出目录符合12层结构规范
- [ ] 文件命名遵循案件编号规则
- [ ] 上下文继承机制完整（引用共享规则）
- [ ] 无重复或冗余内容（引用共享规则而非重复定义）

## 7. 🚫 禁止内容

以下内容**严禁**出现在Subagent配置中：

1. **重复的输出格式示例** - 应引用OutputStandards.md
2. **详细的实现代码** - 属于技能或工具层面
3. **跨Agent的引用链接** - 使用相对路径引用Rules文件
4. **冗长的模板内容** - 使用占位符或引用外部文档
5. **重复的MCP规范** - 应引用PDFProcessingRules.md

## 8. ✅ 最佳实践

### 通用最佳实践

1. **保持简洁** - 每个章节控制在合理长度
2. **结构统一** - 严格按照规范结构组织
3. **引用共享** - 优先引用共享规则而非重复定义
4. **清晰定位** - 明确每个Agent的职责边界
5. **易于维护** - 避免过度复杂的嵌套结构

### SuitAgent特色最佳实践

1. **法律专业性** - 使用准确的法律术语，确保专业性和严谨性
2. **工作流连贯** - 明确定义与其他Agent的协作关系，确保自动触发机制
3. **目录规范** - 严格遵循12层目录结构，文件命名遵循案件编号规则
4. **上下文继承** - 利用`.claude/rules/`中的共享规则，确保输出一致性
5. **质量标准** - 每个Agent都应具备内嵌验证或专项审查机制
